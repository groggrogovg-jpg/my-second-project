Ты работаешь над проектом Neiro Card AI (KardoMatik) на платформе Replit. Необходимо исправить критическую ошибку: новый пользователь, переходя по ссылке, видит сохранённые карточки других пользователей. Также нужно реализовать механизм временного хранения карточек для авторизованных пользователей.

### 1. Привязка карточек к конкретному пользователю

**Проблема:** Карточки не привязаны к пользователю (или привязаны некорректно), из-за чего все видят все карточки.

**Требуемое поведение:**
- Каждый пользователь (включая неавторизованных) должен видеть **только свои** сохранённые карточки.
- Для **неавторизованных** пользователей:
  - При первом посещении создаётся временный идентификатор (сессия) — например, через `localStorage` сгенерировать `sessionId` или использовать `crypto.randomUUID()`.
  - Все карточки, созданные в этой сессии, привязываются к этому `sessionId`.
  - При закрытии браузера или очистке localStorage доступ к карточкам теряется (или они удаляются).
- Для **авторизованных** пользователей (вошли по email/паролю):
  - Карточки привязываются к их `user_id` (из таблицы `users` или `profiles`).
  - При входе карточки из временной сессии (если они есть) должны быть перенесены в постоянный аккаунт (объединены).
  - После входа временная сессия аннулируется.

**Где менять:**
- В логике сохранения карточек (API `/api/cards/save` или подобный) — добавить поле `user_id` (если его нет) и привязывать карточку к текущему пользователю.
- Для неавторизованных — использовать `session_id` вместо `user_id`. Для этого создать в БД таблицу `sessions` или добавить поле `session_id` в таблицу `cards`.
- На фронтенде: при загрузке страницы проверять наличие `sessionId` в localStorage; если нет — генерировать и сохранять.

### 2. Временное хранение карточек для авторизованных (2 дня с предупреждением)

**Требуемое поведение:**
- Для авторизованных пользователей (вошли по паролю) карточки сохраняются **на 2 дня**.
- По истечении 2 дней карточки автоматически удаляются (или архивируются). Для реализации можно добавить поле `expires_at` в таблицу `cards`.
- При входе пользователь видит предупреждение: «Ваши карточки будут храниться 2 дня. По истечении этого срока они будут автоматически удалены. Сохраните их локально, если нужно».
- Предупреждение должно появляться при каждом входе (можно сделать модальное окно, которое закрывается после прочтения).

**Где менять:**
- В таблице `cards` добавить поле `expires_at` (DATETIME) — вычисляется как `NOW() + INTERVAL 2 DAY` при создании карточки.
- На бэкенде создать задачу (cron или проверка при запросе), которая удаляет карточки с истекшим `expires_at`.
- На фронтенде: при успешном входе показывать модальное окно с предупреждением (например, компонент `WarningModal`). Можно использовать localStorage, чтобы не показывать дважды за сессию.

### 3. Миграция данных и проверка

- Если в БД уже есть карточки без привязки к пользователю, их нужно либо удалить, либо присвоить администратору (или создать специальную учётную запись «Гость»).
- После внесения изменений убедись, что новый пользователь видит пустой список карточек.
- Авторизованный пользователь видит только свои карточки, и они имеют срок жизни 2 дня.

### Общие требования:
- Все изменения должны быть совместимы с существующей структурой БД (добавление полей выполнить через миграцию).
- Не ломать существующую логику генерации, балансов и водяного знака.
- После выполнения покажи список изменённых файлов и опиши, как протестировать.

Если структура проекта отличается от предполагаемой, адаптируй изменения под текущую архитектуру.
